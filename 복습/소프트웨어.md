# sw프로세스 모델

- sw프로세스 모델의 정의
    - SDLC(Software Development Life Cycle)
        
        -sw를 어케 개발할 건지 전체적인 흐름을 체계화한 개념
        
        -보기 좋게 깔끔하게 정리하여 볼 것이 목적
        
        -예산과 자원 개발, 관리 방법도 구체적을 정의 가능
        
        -고품질의 소프트웨어 제품 생산을 위함
        
        - 순서
            - 계획
            - 요구사항 정의 및 분석
            - 설계
            - 구현
            - 테스트
            - 유지보수
- 주먹구구식
    - 주먹으로 구구셈하는 것에서 유래
    - 계산없이 대충한다는 의미
    - 간단한 기능만 정리하여 개발
    - 먼저 코드, 제품 만든 후에 요구분석, 설계, 유지보수에 대해 생각
    - 개인이 단시간에 개발할 때 적합(대학 수업 작은 한 학기 프로젝트)
    - 개발단계
        1. 코딩(구현)
        2. 제품생성
        3. 문제가있으면?수정하고 없으면 
        4. 사용
    - 단점?
        - 정확한 문서가 없어 관리 및 유지보수 어려움
        - 대충하는만큼 범위도 모르고 좋은 제품 만들수없
        - 효과적으로 개발 못함
        - 수정을 계속하다보면 프로그램 구조 엉킬수도 있음
- 선형 순차적 모델
    - Linear sequential
    - waterfall(폭포수 모델)
        - 장점
            - 관리가 용이
            - 체계적인 문서화
            - 요구사항의 변화가 적은 프로젝트에 적합
        - 단점
            - 단계가 정확히 정해져 있어 전단계가 완벽히 수행된 후 넘어감
            - 중간 결과를 볼 수 없어 답답함
    - classic life cycle
    - v모델
        - 폭포수 모델+테스트 단계 추가 확장
        - 개발 단계 검증(테스트)하는 데 초점
- 진화적 프로세스 모델
    - 프로토타입 모델
        - 조금조금씩 단계나눠 만들고 사용자 만족도 묻고 수정하고 이를 반복
        - 프로토타입은 사용자의 요구 받기위한(의사소통) 테스트 모형
        
        ---
        
        - 프로토타입의 생명주기
            - 요규사항 정의 및 분석 안에서 활개
                
                -요구수정
                
                -빠른 설계
                
                -프로토타입 개발
                
                -고객평가
                
                -프로토타입 조정
                
                -구현
                
        - 실험적 프로토타입 모델
            
            위와 단계흐름은 비슷하지만 고객평가에서 좋지 않은 반응있으면 수정요구사항 받들고 다시 설계, 좋으면 바로 최종 사용자 요구 받들고 분석, 설계, 구현까지 쭉쭉 진행
            
        - 진화적 프로토타입 모델
            
            얘는 프로토타입 만들고 그걸로 평가 받음 좋으면 최종제품 아니면 수정 요구사항 받들고 다시 설계하고 개발
            
        - 개발절차
            1. 요구사항 정의 및 분석
            -처음 요구사항 정의한담에 계속 반복하며 개발
            2. 프로토타입 설계
            -완전한 개발대신 사용자가 써볼수있을 정도로 설계
            -입출력 화면을 통해 사용자 인터페이스 중심 설계
            3. 프로토타입 개발
            -완제품까진 바라지 않는다
            -입출력 통해 사용자 요구 항목만 보면 된다
            -출력 결과 보고 사용자가 원하는 것인지 확인
- 통합 프로세스(UP) 모델
    - 비즈니스 모델링
        - 개발 전 비즈니스 환경 이해하고 분석하는 작업
        - 시스템 목적(비즈니스 문제) 및 범위 정의
        - 실제 업무에서 사용하는 개념을 시각화하여 도메인 모델 구축
        - 조직 내 수행되는 업무 절차를 흐름도나 BPMN등으로 시각화
        - 주체가 어떤 작업을 수행하는지 분석
        - 프로젝트의 기반 제공
        - 아까 도메인 모델에서 정의한 용어와 개념이 클래스 구조를 설계하는데 직접적을 활용
        
        ---
        
        - BPMN
            
            조직내의 업무 흐름(비즈니스 프로세스)를 시각화하고 표준화하기 위한 글로벌 그래픽 표기법
            
    - 방법
        1. 먼저 비즈니스 모델링과 함계 초기계획, 프로젝트 계획을 세움
        2. 다음에 아까의 방식과 같이 순서 순환돌며 최선의 결과물 만들어내서
        3. 배치
    - 절차
        1. 도입 단계
        2. 구체화 단계
        3. 구축 단계
        4. 전이 단계
        5. 도입/구체화/구축전이 단계의 공통 작업
        -형상(변화) 관리
        -프로젝트 관리
        -환경 점검
- 애자일 프로세스 모델
    - 고객 불편한점 즉각 수용하여 문제 풀어나가는 방법
    - 개개인(고객)과 소통 중시
    - 실행 가능한 소프트웨어 중시
    - 계획보단 변화
    
    ---
    
    - 개발 방법
        
        반복적인 개발을 통해 잦은 출시 목표
        
        -프로토타입 개발→사용자 확인→일부기능사용
        
        ---
        
        - 스크럼 개발 프로세스
            - 개발보단 팀 개선과 관리 위함
            - 경험적 관리 기법 중 하나
            - 구체적인 프로세스 명확하게 제시 놉
            - 팀을 운영할 때 아주 좋은 방식
            - 똘똘 뭉쳐서 짧은 구간 ***전력질주(sprint)**하여* 목표 달성!!
